# 19. What Will Be Compared

In the forthcoming labs and case studies, each comparison should be deliberate—not a random pair of prompts. The list below is what you will eventually walk through, item by item, in exercises and in write-ups. It targets request structures that people actually use, misuse, or half-use in real work.

## Prompt families to compare

The forthcoming labs will focus on contrasts such as:

### 1. Vague request vs. structured request

Example:

- “Look at this and improve it.”
- a version with goal, scope, constraints, and definition of done

### 2. Surface fix request vs. complete-delivery request

Example:

- “Fix the bug.”
- a version that requires root cause, hidden issue review, and verification

### 3. Implementation-first request vs. design-first request

Example:

- “Build this.”
- a version that first requires alternatives, trade-offs, and change axes

### 4. Review summary request vs. risk-first review request

Example:

- “Review this code.”
- a version that explicitly prioritizes bugs, regressions, edge cases, and missing tests

## Why these comparisons matter

These prompt families map directly onto the theory of the book. They are not decorative examples. They are practical contrasts between weak task framing and stronger task framing.

The point is not to prove that “longer prompts win.” The point is to show that **better-structured work requests produce different kinds of work**.
