# 4. 目标、约束与完成定义

如果您想让 AI 做出更好的工作，有三个东西比大多数人想的更重要：

- 目标
- 约束
- 完成定义

缺少它们时，再聪明的模型也像是在雾里工作。

## 目标：这项工作真正是为了什么

目标不是动作本身。“重构这个模块”是动作；“把这个模块重构到以后新增支付渠道时不必改五个无关类”才是目标。

这区别非常重要，因为目标会帮助模型做权衡。当请求不完整时，模型往往会优化一种“泛泛有用性”；而一旦目标明确，模型才能开始优化真正相关的东西。

例如：

```text
改进这个函数。
```

不如：

```text
改进这个函数，让它更容易测试、更容易扩展新的提供方，并减少它对全局状态的依赖。
```

第二种说法已经把“哪些改进更重要”写进去了。

## 约束：哪些东西不能被牺牲

很多糟糕的 AI 结果，不是因为模型不够强，而是因为约束没写清。

约束的作用，是告诉模型：在改变这项工作时，哪些东西必须保持稳定。常见约束包括：

- 不改变公开行为
- 不显著增加延迟
- 不破坏现有 API
- 不做整模块重写
- 不引入新依赖

约束不是多余的形式，而是保持问题形状的方式。

## 完成定义：这项工作什么时候才能停

很多认真用户恰恰在这里写得最弱。

任务如果带有探索性，模型可能会无限扩张；任务如果定义太弱，模型又会太早停下。完成定义就是用来同时解决这两个问题的。

一个强完成定义通常会回答：

- 什么必须被找到
- 什么必须被修改
- 什么必须被检查
- 什么必须被汇报

例如：

```text
完成定义：
1. 已确认根因
2. 当前问题已修复
3. 已检查相关隐藏风险
4. 已完成验证
5. 已说明剩余风险
```

这比一句“请彻底一点”强得多。

## 为什么这三件事会直接改变模型行为

清晰目标会收窄相关性。  
清晰约束会保护边界。  
清晰完成定义会阻止过早收尾。

它们放在一起，会直接改变模型如何分配注意力和工作量：

- 更倾向于先调查再回答
- 不容易被“看起来有用但其实不相关”的改动吸走
- 更可能在合适的地方停，而不是在方便的地方停

## 一个最值得复用的基础骨架

如果您只升级一件事，那就升级这个结构：

```text
目标：

约束：
1.
2.
3.

完成定义：
1.
2.
3.
```

这是整个 AI 协作里最值得建立的高杠杆习惯之一。

## 收束

很多人以为“更会提问”意味着更会修辞。其实大多数严肃任务里，真正重要的是更会设计工作。目标、约束和完成定义，就是那套工作设计的核心。
