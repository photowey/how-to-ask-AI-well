# 13. 遗留代码重构

遗留代码最容易把 AI 协作拉向两个极端：一种是过度保守，只做表面整理；另一种是过度激进，一上来就想推倒重来。模糊提问往往会在这两个极端之间来回摆动。

更好的提问方式，是逼着模型走一条中间路线：既不只擦边修饰，也不脱离现实约束乱开大刀。

## 在遗留代码里，真正重要的是什么

大多数遗留重构的核心目标并不是“更美”，而是：

- 职责更清楚
- 耦合更低
- 更容易测试
- 后续变化更便宜

这意味着提问必须强迫模型区分：

- 表面清理
- 结构改善
- 当前不处理但仍然残留的技术债

## 一个更合适的重构提问

```text
这是一个重构任务，不是表层清理任务。
请围绕可维护性、可测试性和未来扩展能力，改进这个模块的结构。
不要提出整模块重写。
请先识别主要结构问题，再给出重构方向，最后说明还有哪些债务残留。
```

## 需要尽量提供的上下文

- 模块的职责
- 当前最痛的点
- 常见变更场景
- 已知耦合点和测试问题

如果这些信息缺失，模型就很容易优化“泛泛的整洁感”，而不是您真正想降低的未来改动成本。

## 什么样的结果算重构有价值

一个好的遗留重构结果通常包含：

- 更清楚的职责边界
- 更少的隐藏依赖
- 更稳定的测试切入点
- 对“哪些问题这次改了、哪些还没改”给出诚实说明

## 本章小结

和 AI 一起做遗留重构，最重要的不是追求“焕然一新”，而是在约束中换来更好的结构。模糊提问会得到表面化的整理，好提问才有机会得到真正降低未来成本的重构。
