# 9. 可复用的提问模式

模式之所以重要，是因为真实工作会重复发生。您不可能每次遇到 bug、审查、设计讨论或深度排查时，都从零构建一套提问结构。

但可复用模式真正有用的前提是：它们必须是“结构化工作框架”，而不是漂亮措辞。好的模式不是一大段戏剧化文本，而是一套能承载真实任务的紧凑骨架。

## 模式一：完整交付任务

当您不想要浅层收尾时，用这个：

```text
把这项任务按完整交付来处理。
先扫描完整的相关范围。
在当前范围内尽量一次性收口。
不要把还没做完的工作甩成“下一步建议”。
只有遇到真实阻塞时才允许停下。
```

## 模式二：深度排障

当您已经看见症状，但真正原因还不清楚时，用这个：

```text
这是一个深度优先的排障任务。
不要停在表面症状上。
请找到根因，检查直接相关的上下游组件，并说明这个修复是如何被验证的。
如果还存在隐藏的相关风险，也请一并指出。
```

## 模式三：先设计，再实现

当任务本身还带有设计不确定性时，用这个：

```text
先不要实现。
请先给出 2 到 3 套设计方案，说明边界、权衡、变化轴，以及未来演进成本。
在进入实现之前，先推荐一个方案并解释理由。
```

## 模式四：风险优先代码审查

当您真正想要的是发现问题，而不是礼貌总结时，用这个：

```text
请围绕 bug、行为回归、边界情况、设计风险和缺失测试来审查这次改动。
不要优先给风格意见。
如果没有明确发现，也请直接说明，并补充剩余风险。
```

## 模式五：通过变化轴要求更好的设计

当“做得更好一点”太空泛时，用这个：

```text
未来变化轴：
1.
2.
3.

请提出一种设计，让这些未来变化的成本变低，同时又不要把当前版本过度设计。
```

## 正确使用模式的方法

不要机械地复制粘贴。真正有效的做法是把模式里的抽象词替换成真实任务：

- 用真实模块、仓库、系统替换“范围”
- 用真实约束替换“约束”
- 用真实验证手段替换“验证”
- 用您真正接受的完成标准替换“完成定义”

## 收束

模式存在的价值，不是让您“看起来更会用 AI”，而是防止您在时间压力下反复漏掉那些最重要的结构件。
